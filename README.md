# Asterum Integrated Scheduler

아티스트와 제작진이 함께 사용하는 통합 스케줄 관리 시스템입니다.  
일회성 일정부터 복잡한 반복 일정까지, 팀 단위 참여자 지정과 리소스 중복 방지까지 지원합니다.

---

## 실행 방법

### 사전 요구사항

- Java 25+
- Node.js 18+
- npm

### 백엔드 실행

```bash
cd backend

# macOS / Linux
./gradlew bootRun

# Windows
gradlew.bat bootRun
```

백엔드는 `http://localhost:8083` 에서 실행됩니다.

> H2 콘솔: `http://localhost:8083/h2-console`  
> JDBC URL: `jdbc:h2:file:./data/asterum`  
> H2 DB 로그인: `sa` || `비밀번호없음`

### 프론트엔드 실행

```bash
cd frontend
npm install
npm run dev
```

프론트엔드는 `http://localhost:5173` 에서 실행됩니다.  
API 요청은 자동으로 `http://localhost:8083` 으로 프록시됩니다.

### 초기 데이터

백엔드 최초 실행 시 아래 시드 데이터가 자동으로 생성됩니다.

| 유형 | 데이터 |
|------|--------|
| 멤버 | 5명 (예준, 노아, 은호, 밤비, 하민) |
| 스태프 | 4명 (카메라맨A, 편집자B, 디자이너C, 음향기사D) |
| 팀 | 3팀 (영상팀, 디자인팀, 프로덕션팀) |
| 리소스 | 3개 (메인 스튜디오, B 세트장, 회의실 A) |

## 구현된 기능

### MVP 1: 일회성 일정 관리

- 월간 달력에서 날짜를 클릭해 일정 등록
- 제목, 날짜, 시작/종료 시간 입력
- 참여자 선택 (멤버 또는 스태프 개별 지정)
- 촬영 장소(리소스) 선택
- 등록된 일정 클릭 시 상세 확인, 수정, 삭제

### MVP 2: 반복 일정 관리

**반복 옵션**
- 매일 / 매주 / 매월 반복

**종료 조건**
- 특정 날짜까지 반복
- 특정 횟수 반복
- 무기한 반복

**범위 지정 수정/삭제**
- `이 일정만` — 해당 날짜 1건만 변경
- `이후 모든 일정` — 선택일 포함 이후 전체 변경
- `전체 일정` — 반복 일정 전체 변경

**예외 처리**
- 반복 일정 중 특정 날짜만 별도 수정 가능 (예외 일정)

### 선택 기능

| 기능 | 구현 여부 | 설명 |
|------|-----------|------|
| 유형 전환 | ✅ | 일회성 일정을 반복 일정의 시작점으로 즉시 전환 |
| 팀 단위 지정 | ✅ | 개별 참여자 대신 팀 전체를 참여자로 지정 |

| 영속성 유지 | ✅ | H2 파일 모드로 서버 재시작 후에도 데이터 보존 |

### 추가 기능

- **월간 뷰**: 달력 그리드, 날짜별 일정 목록, 오버플로우 처리
- **주간 타임라인 뷰**: 시간축 기반 주간 타임라인으로 시간대별 일정 시각화
- **참여자 관리**: 멤버·스태프 추가·편집·삭제
- **팀 관리**: 팀 생성·편집·삭제, 팀원 구성 관리

## DB 연관관계

```
participants ─────────────────────────────────────────────────────────┐
     │                                                                │
     │ (N:M)                                                          │ (N:M)
     │                                                                │
schedule_series ──── (1:N) ──── schedule_occurrences                  │
     │                                    │                           │
     │ (N:M)                              │ (N:M)                     │
     │                                    │                           │
  teams ◄──── team_members ────► participants                         │
     │                                                                │
     └──── schedule_series_teams ◄────────┘                           │
                                                                      │
schedule_series ──────────────────────────────────────────────────────┘
     │
     └── resource_id (FK, nullable) ──► resources
```

**핵심 관계 요약**

| 관계 | 설명 |
|------|------|
| `schedule_series` → `schedule_occurrences` | 1:N, 반복 정의 → 개별 일정 항목 |
| `schedule_series` ↔ `participants` | N:M, 반복 일정 기본 참여자 |
| `schedule_occurrences` ↔ `participants` | N:M, 예외 발생 시 덮어쓰기 |
| `schedule_series` ↔ `teams` | N:M, 팀 단위 참여 |
| `schedule_occurrences` ↔ `teams` | N:M, 예외 발생 시 덮어쓰기 |
| `teams` → `participants` | N:M (`team_members` 조인) |
| `schedule_series` → `resources` | N:1, 장소 지정 (선택) |
| `schedule_occurrences` → `resources` | N:1, 예외 발생 시 덮어쓰기 |

## 테이블 설명

### `participants`
아티스트 멤버 및 스태프를 저장합니다.

| 컬럼 | 설명 |
|------|------|
| `id` | PK |
| `name` | 참여자 이름 |
| `type` | `MEMBER` (멤버) / `STAFF` (스태프) |
| `created_at`, `updated_at` | 생성/수정 시각 |

---

### `teams`
팀 단위 참여자 그룹을 저장합니다.

| 컬럼 | 설명 |
|------|------|
| `id` | PK |
| `name` | 팀 이름 (예: 영상팀) |

---

### `team_members`
팀과 참여자 간의 N:M 조인 테이블입니다.

| 컬럼 | 설명 |
|------|------|
| `team_id` | FK → `teams.id` |
| `participant_id` | FK → `participants.id` |

---

### `resources`
스튜디오, 세트, 연습실 등 촬영 장소를 저장합니다.

| 컬럼 | 설명 |
|------|------|
| `id` | PK |
| `name` | 장소명 (예: 스튜디오 A) |
| `category` | `STUDIO` / `SET` / `ROOM` |

---

### `schedule_series`
반복 일정의 **정의(definition)** 를 저장합니다. 일회성 일정도 내부적으로 `recurrence_type = null` 인 반복 정의로 저장됩니다.

| 컬럼 | 설명 |
|------|------|
| `id` | PK |
| `title` | 일정 제목 |
| `start_time` | 시작 시각 |
| `end_time` | 종료 시각 |
| `resource_id` | FK → `resources.id` (nullable) |
| `recurrence_type` | `DAILY`(매일) / `WEEKLY`(매주) / `MONTHLY`(매월) / null(반복 없음) |
| `interval_value` | 반복 간격 (예: 2이면 2주마다) |
| `end_type` | `UNTIL_DATE`(날짜까지) / `COUNT`(횟수까지) / `NEVER`(무기한) |
| `until_date` | 반복 종료 날짜 (end_type = UNTIL_DATE) |
| `occurrence_count` | 반복 횟수 (end_type = COUNT) |
| `anchor_date` | 반복의 기준 시작 날짜 |
| `active` | 활성 여부 |

---

### `schedule_occurrences`
반복 일정으로부터 생성된 **개별 일정 항목**을 저장합니다.  
예외 처리(수정/삭제)가 발생한 경우 해당 일정 항목만 별도 데이터를 보유합니다.

| 컬럼 | 설명 |
|------|------|
| `id` | PK |
| `series_id` | FK → `schedule_series.id` |
| `title` | 개별 수정된 제목 (예외 시) |
| `occurrence_date` | 해당 일정의 날짜 |
| `start_time` / `end_time` | 개별 수정된 시각 (예외 시) |
| `resource_id` | 개별 수정된 장소 (예외 시) |
| `status` | `ACTIVE`(활성) / `CANCELLED`(취소됨) |
| `is_exception` | 반복 일정 기본값에서 변경된 예외 여부 |

---

### `schedule_series_participants` / `schedule_occurrence_participants`
반복 일정 또는 개별 일정 항목과 참여자 간의 N:M 조인 테이블입니다.  
예외 처리된 일정 항목은 `occurrence_participants`에서 참여자를 개별 덮어씁니다.

---

### `schedule_series_teams` / `schedule_occurrence_teams`
반복 일정 또는 개별 일정 항목과 팀 간의 N:M 조인 테이블입니다.

## 주요 설계 결정

**일정 모델: Series(반복 묶음) + Occurrence(개별 일정 항목) 분리**  
반복 일정을 `schedule_series`(반복 일정 묶음 정의)와 `schedule_occurrences`(개별 일정 항목)로 분리해 저장합니다. 일회성 일정도 동일 모델을 공유하며, 예외 처리(특정 날짜만 수정)는 해당 개별 일정에만 데이터를 덮어쓰는 방식으로 구현됩니다.

**리소스 중복 방지**  
일정 생성/수정 시 동일 날짜·시간에 동일 리소스가 이미 사용 중인지 DB 레벨에서 검증합니다.

**에러 응답 형식**  
RFC 9457 표준 오류 응답 형식을 따릅니다 (`type`, `title`, `status`, `detail`).

## AI 활용

### 1. AI 활용 범위

과제 전반에 걸쳐 코딩 에이전트(codex)를 활용했습니다. 다만 처음부터 하네스 엔지니어링을 위한 환경(규칙 문서, 스킬 정의 등)을 사전에 구성하지 않고 진행했습니다. AI 도구를 능숙하게 세팅해 결과물만 뽑아내는 것보다, 요구사항을 분석하고 설계를 검토하며 문제를 해결해나가는 과정에서 AI를 어떻게 활용하는지 보여드리는 것이 이번 과제의 목적이라고 판단했기 때문입니다.

**요구사항 분석 및 일정 계획**  
요구사항을 받은 직후 AI와 함께 기능 단위 분석을 먼저 진행했습니다. 단순 기능 파악에서 끝나지 않고, 마감일보다 3일 앞서 완료할 수 있도록 현실적인 TASK 단위 일정을 함께 도출했습니다.  
(예: 퇴근 후 작업 가능 시간 기준, 일요일은 최대 집중 작업일로 배정)

**코드 생성 및 구조 설계**  
MVP 기능 구현 전반에 걸쳐 코딩 에이전트를 활용했습니다. 초기 API 설계, 도메인 구조, DB 스키마 초안을 함께 작성하고, 이를 검토하며 개선해나가는 방식으로 진행했습니다.

**프론트엔드 UI 개선**  
기본 디자인이 밋밋하다고 판단해 에이전트 ui-ux-pro-max 스킬을 이용해 디자인을 개선했습니다. 컨셉은 싱글 앨범 BBUU와 함께 벚꽃이라는 테마를 잡아 디자인했습니다.  
참여자 등록 캐릭터는 플레이브 BBUU 앨범 커버를 이용했으며, OpenAI의 이미지 생성 모델인 `덕테이프`를 이용해 도트 캐릭터를 제작한 뒤, 추출한 이미지를 SVG로 변환하여 활용했습니다. 또한 agentation 라이브러리를 사용해 선택된 UI 태그를 에이전트에 정확히 전달함으로써 UI 수정을 효율적으로 처리했습니다.

### 2. 주요 논의 내용: 수용과 반려

**[기술 스택 선정 — 반려]**  
에이전트는 Kotlin 사용을 제안했으나 Java를 선택했습니다. Kotlin 문법은 어느 정도 알고 있지만, "코틀린스럽게" 작성하는 부분이 아직 충분하지 않다고 생각했습니다. 코딩 에이전트가 생성한 코드를 제한된 시간 내에 온전히 이해하고 검토하기 어렵다고 판단했고, 이는 품질 리스크로 이어질 수 있어 익숙한 Java를 선택했습니다.

**[동시성 처리 설계 — 부분 수용 후 구체화]**  
에이전트가 제안한 초기 설계는 낙관적 락(여러 요청이 동시에 들어올 때 충돌을 나중에 감지하는 방식)으로 시간대 충돌을 방지하는 방식이었습니다. 방향성은 수용했지만 구현이 추상적이었습니다. `@Lock(LockModeType.PESSIMISTIC_WRITE)`, `SELECT FOR UPDATE`를 명시적으로 사용하도록 제약 조건을 구체화하여 실제 동시 접근 제어 로직을 보완했습니다.

**[반복 일정 설계 — 반려 후 재설계]**  
에이전트는 "1년치를 미리 생성하고, 필요 시 추가 생성"하는 방식을 제안했습니다. 그러나 '필요 시'가 언제인지, 중복 생성은 어떻게 방지하는지 등 핵심 기준이 정의되지 않아 실 운영에서 문제가 발생할 수 있다고 생각했습니다. 이에 반복 조건, 추가 생성 시점, 중복 방지 로직을 명확히 재정의한 후 구현 방향을 새로 수립했습니다.

**[단건 수정 처리 — 반려 후 논리 삭제 방식 채택]**  
에이전트가 참/거짓 표시(boolean flag) 형태로 단건 수정을 처리하려 했으나, 실제 데이터를 DB에서 지우지 않고 '삭제됨' 여부만 표시하는 논리 삭제 방식으로 처리하도록 구체적인 방향을 제시하고 변경했습니다.

**[시드 데이터 구성 및 참여자 관리 기능 — 반려 후 확장]**
초기 설계에서는 시드 데이터를 아티스트와 스태프만 포함하는 형태로 구성했습니다. 그러나 실제 운영 환경을 상상해보았을 때, 아티스트 멤버는 고정이더라도 스태프와 팀은 언제든 새로 생기거나 변동될 수 있다는 점이 눈에 걸렸습니다. 시드 데이터만으로는 이 변화에 대응할 수 없다고 판단하여, 스태프와 팀을 직접 추가·수정·삭제할 수 있는 관리 기능을 추가로 구현했습니다.

### 3. 검토 및 수정 과정

**[Map 기반 요청/응답 → DTO 객체로 전환]**  
초기 MVP 구현 후 코드 검토 중, 요청과 응답이 `Map`으로 처리되고 있는 것을 확인했습니다. 타입 안전성이 없고 유지보수가 어렵다고 판단해, 요청과 응답 각각을 DTO 객체로 분리하여 재구성했습니다.

**[확장 기능 구현 시 기존 구조 훼손 문제]**  
확장 기능 추가 단계에서 코딩 에이전트가 기능 구현에만 집중하다 보니 기존 구조를 임의로 변경하는 문제가 발생했습니다. 이를 방지하기 위해 `AGENTS.md` 규칙 문서를 정의하여 현재 구조에서 벗어나지 못하도록 제약했습니다. 또한 규칙 내용이 길어질수록 에이전트 효율이 저하되는 점을 고려해, `README.md`에 Index를 걸어 참조하도록 구성했습니다.

**[서비스 레이어 비대화 문제]**  
서비스 클래스에 너무 많은 책임이 집중되는 문제가 발생했습니다. 패키지 구조를 `Presentation / Application / Domain / Infrastructure` 형태로 리팩토링하고, 스케줄의 반복 유형(매일·매주·매월)이나 유효성 검증 로직은 도메인 레이어에 위치시키고, 서비스 레이어는 이를 조립하는 역할만 담당하도록 책임을 분리했습니다.

### 끝으로..
이번 과제를 진행하면서 단순히 동작하는 기능을 만드는 것에 그치지 않고, 실제로 이 시스템을 쓸 사람들을 계속 떠올리며 만들었습니다. 촬영 현장에서 기술팀, 디자인팀, 등 아티스트가 하나의 캘린더를 보며 빠르게 소통할 수 있는 시스템이라면 어떤 모습이어야 할까를 고민했고, 그 고민이 팀 단위 참여자 지정이나 장소 중복 방지 같은 설계 결정으로 이어졌습니다.
또 한편으로는, 플레이브의 팬인 플리들이 직접 쓸 수 있는 수준의 스케줄러 앱이 있다면 어떨까 하는 상상도 했습니다. 그래서 기능 명세를 충족하는 것에서 나아가 디자인 컨셉과 사용자 경험까지 함께 고민했습니다. 아이디어는 많았지만 프로토타입 과제라는 특성상 모두 구현하지 못한 부분은 솔직히 아쉬움으로 남습니다.
그럼에도 이 과제가 단순한 코드 제출이 아닌, 제가 어떤 시각으로 서비스를 바라보고 어떻게 문제를 풀어가는지 보여드릴 수 있는 기회가 되었기를 바랍니다.
